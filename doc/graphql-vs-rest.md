# GraphQL은 왜 쓰는가, REST API와의 차이점

## GraphQL을 쓰는 이유

- **오버페칭/언더페칭 방지**: REST는 엔드포인트가 정해준 응답 구조를 그대로 받아야
  한다. 필요한 필드가 3개뿐이어도 서버가 내려주는 전체 객체를 다 받거나(오버페칭),
  반대로 화면 하나 그리는 데 여러 엔드포인트를 조합 호출해야 한다(언더페칭).
  GraphQL은 클라이언트가 쿼리에 명시한 필드만 정확히 받는다.
- **단일 엔드포인트**: REST는 리소스마다 `/posts`, `/posts/{id}`, `/posts/{id}/comments`
  처럼 엔드포인트가 늘어난다. GraphQL은 `/graphql` 하나로 받고, 무엇을 조회/변경할지는
  쿼리 문서(operation)가 결정한다.
- **스키마가 곧 계약이자 문서**: 타입/필드가 스키마(`schema.graphqls`)에 정의되고,
  introspection으로 클라이언트가 스키마를 직접 조회할 수 있다. Swagger처럼 별도
  스펙 파일을 유지보수할 필요가 없다 ([[graphiql-usage]] 참고).
- **여러 리소스를 한 번에 조합 조회**: 예를 들어 게시글 + 작성자 + 최근 댓글 3개를
  화면 하나에 그려야 할 때, REST면 API 3번 호출 후 클라이언트가 조합해야 하지만
  GraphQL은 쿼리 하나로 트리 형태로 받을 수 있다.

## REST API와의 주요 차이점

| 구분 | REST | GraphQL |
|---|---|---|
| 엔드포인트 | 리소스별로 여러 개 (`/posts`, `/posts/{id}`) | 단일 엔드포인트 (`/graphql`) |
| 응답 형태 | 서버가 고정 (필드 선택 불가) | 클라이언트가 쿼리로 필요한 필드만 선택 |
| HTTP 메서드 | GET/POST/PUT/DELETE로 의미 구분 | 항상 POST 하나, 대신 Query(조회)/Mutation(변경)로 구분 |
| 버저닝 | `/v1/`, `/v2/` 처럼 URL/헤더로 버전 관리 | 필드 단위로 점진적 추가/deprecated 처리, 버전 개념 자체가 약함 |
| 문서화 | Swagger/OpenAPI 같은 별도 스펙 필요 | 스키마 자체가 문서, introspection으로 실시간 조회 (GraphiQL) |
| 캐싱 | HTTP 캐싱(URL 기반)을 그대로 활용 가능 | 응답 구조가 매번 달라질 수 있어 URL 기반 캐싱이 어렵고 별도 캐싱 전략 필요 |
| 에러 처리 | HTTP 상태 코드로 표현 (404, 400 등) | 대부분 200 OK + `errors` 배열, 상태 코드에 덜 의존 |
| 러닝 커브 | 낮음, HTTP만 알면 됨 | 스키마 설계, 리졸버 개념 등 추가 학습 필요 |

## 이 프로젝트에서의 적용

이 프로젝트는 헥사고날 아키텍처를 쓰기 때문에 REST든 GraphQL이든 인바운드 어댑터
(`adapter.in.graphql`)만 갈아끼우면 된다. 즉 GraphQL을 선택한 것은 API 스타일의
문제일 뿐, `domain`/`application` 계층은 REST로 바꿔도 전혀 영향받지 않는다
(자세한 구조는 프로젝트 루트 `CLAUDE.md` 참고).
